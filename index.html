재고, 발주, 정산 등 복잡한 물류 데이터를 관리하고, 경영진의 빠른 의사결정을 돕는 대시보드를 제공하는 시스템입니다.

👉 1. 대시보드 데이터 집계 성능 최적화 (N+1 문제 해결)

[문제] 초기에는 1월부터 12월까지의 매출/지출 데이터를 각각 조회하는 루프로, 페이지 로딩 시 총 24번의 쿼리가 실행되어 2~3초의 로딩 지연이 발생했습니다.

[해결] MyBatis에서 GROUP BY를 사용하여 연간 데이터를 단 2번의 쿼리로 조회하고, 백엔드에서 Stream API로 데이터를 가공하여 쿼리 횟수를 92% 감소시켰습니다. 

<select id="selectYearlySales" resultType="SalesVO">
    SELECT MONTH(sales_date) as month, SUM(amount) as totalAmount
    FROM sales
    WHERE YEAR(sales_date) = #{year}
    GROUP BY MONTH(sales_date)
</select>

[결과] 로딩 속도를 0.4초로 단축하여 사용자 경험을 개선했습니다.

👉 2. 성장률(MoM/YoY) 계산 로직의 안정성 확보

[문제] 전월 매출이 '0'일 경우, 성장률 계산 시 ArithmeticException (Division by Zero)이 발생했습니다.

[해결] 분모가 0인지 확인하는 삼항 연산자 방어 로직을 추가하여 예외 발생을 원천 차단했습니다. 

double growthRate = (lastMonthAmount == 0) 
    ? (currentMonthAmount > 0 ? 100.0 : 0.0) 
    : ((double)(currentMonthAmount - lastMonthAmount) / lastMonthAmount) * 100;

[결과] 데이터가 불충분한 상황에서도 에러 없이 안정적으로 대시보드를 표출합니다.

👉 3. 전표 번호 생성의 동시성 제어

[문제] 매출/지출 등록 시 고유 전표 번호 생성 과정에서, 다중 요청 시 중복 번호가 발생할 가능성이 있었습니다.

[해결] 저장, 포맷팅, 갱신 과정을 @Transactional으로 묶어 원자성을 보장했습니다. 

@Transactional(rollbackFor = Exception.class)
public void registerSales(SalesDTO salesDTO) {
    // 1. 기본 정보 저장 (ID 생성)
    salesMapper.insertSales(salesDTO);
    // 2. 생성된 ID로 고유 전표 번호 포맷팅
    String serialNo = generateSerialNo(salesDTO.getId());
    // 3. 전표 번호 업데이트 (원자성 보장)
    salesMapper.updateSerialNo(salesDTO.getId(), serialNo);
}

[결과] 데이터 정합성을 유지하여 안정적인 전표 관리 시스템을 구축했습니다.

게스트하우스 플랫폼: AI 호스트 & 서버 아키텍처

Spring Boot, Vue.js, Google Gemini, Docker, Nginx

AI가 숙소 리뷰를 분석하여 개선점을 제안하고, 이중화된 서버 아키텍처로 안정성과 보안을 강화한 플랫폼입니다.

👉 1. AI 응답 불확실성 및 데이터 파싱 오류 해결

[문제] Gemini API가 `List<String>`이 아닌 단순 `String`이나 포맷이 깨진 데이터를 간헐적으로 반환하여 `ClassCastException`이 발생했습니다.

[해결] AI 응답을 `Object`로 받은 뒤, `instanceof`로 타입을 확인하고 다양한 케이스(List, JSON 문자열, 쉼표 분리 문자열)에 대응하여 `List<String>`으로 안전하게 변환하는 2단계 방어적 파싱 로직(`parseSafe`, `convertToList`)을 구현했습니다.

[결과] 파싱 에러율 0%를 달성하여 AI 기능의 안정성을 확보했습니다.

👉 2. Nginx 502 Bad Gateway (리버스 프록시 라우팅 실패)

[문제] 서버 분리 후, Public Subnet의 Nginx가 Private Subnet에 있는 Admin Server(`10.0.x.x:8081`)를 찾지 못해 502 에러가 발생했습니다.

[해결] Nginx의 `location /api/admin/` 블록에 `proxy_pass` 대상으로 `localhost`가 아닌 Admin 서버의 Private IP를 명시하여 라우팅 경로를 확정했습니다. 

# /etc/nginx/sites-available/default
location /api/admin/ {
    # Admin 서버의 Private IP를 직접 지정
    proxy_pass http://10.0.x.x:8081;
    proxy_set_header Host $host;
}

[결과] 외부 요청이 Nginx를 통해서만 내부망 서버에 도달하도록 강제하여 보안을 강화했습니다.

👉 3. Docker "Bind for 0.0.0.0:8080 failed" (좀비 포트 충돌)

[문제] `docker run` 시 8080 포트가 이미 사용 중이라는 에러가 발생했으며, `netstat`으로 확인해도 점유 프로세스가 보이지 않는 ''Ghost Port'' 현상이었습니다.

[해결] `systemctl stop docker.socket`으로 소켓을 정리하고, User/Admin 서버의 Host 포트를 각각 `8080`, `8081`로 명확히 분리하여 충돌을 원천적으로 방지했습니다. 

# User Server
docker run -d -p 8080:8080 --name user-server your-image

# Admin Server
docker run -d -p 8081:8081 --name admin-server your-image

[결과] 안정적인 컨테이너 배포 환경을 구축했습니다.

Dynamic Beat: Java 멀티스레드 리듬 게임

Java, Swing, Multi-threading, AWT

Java의 핵심 CS 개념인 멀티스레딩, 동기화, GUI 렌더링을 학습하기 위해 제작한 리듬 게임입니다.

👉 1. 수많은 노트 스레드로 인한 메모리 누수 해결

[문제] 게임이 진행될수록 생성된 Note 스레드가 누적되어 JVM 힙 메모리 사용량이 급증하고 렉이 발생했습니다.

[해결] 판정선을 지나거나 키 입력이 된 노트는 즉시 `interrupt()`를 호출하여 스레드를 종료시키고, `ArrayList`에서 해당 객체를 `remove()`하여 GC가 자원을 수거하도록 유도했습니다.

[결과] 장시간 플레이 시에도 안정적인 메모리 점유율을 유지했습니다.

👉 2. 음악과 노트의 싱크 밀림 현상 (Time Drift)

[문제] `Thread.sleep()`의 오차로 인해 시간이 갈수록 음악보다 노트가 느려지는 현상이 발생했습니다.

[해결] `sleep`에만 의존하지 않고, Music 스레드에서 현재 재생 중인 음악의 타임스탬프를 실시간으로 가져와 `(현재시간 - 노트생성시간) * 속도` 공식을 적용하여 노트의 Y좌표를 강제로 동기화했습니다.

[결과] 곡이 끝날 때까지 1ms 오차 범위 내에서 정확한 싱크를 유지했습니다.

👉 3. 화면 깜빡임 (Screen Flickering) 현상

[문제] AWT의 `paint()` 메소드가 캔버스를 지우고 다시 그리는 과정이 눈에 보여 심한 깜빡임이 발생했습니다.

[해결] 가상 이미지를 생성하여 모든 그래픽 요소를 그린 뒤, 완성된 이미지를 한 번에 출력하는 **더블 버퍼링(Double Buffering)** 기법을 도입했습니다.

[결과] 60fps 수준의 부드러운 애니메이션을 구현했습니다.
